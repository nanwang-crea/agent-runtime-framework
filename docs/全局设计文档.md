# 全局设计文档

## 1. 总体目标

`agent-runtime-framework` 的长期目标不是做一个单一桌面文件助手，而是构建一个可扩展的桌面 AI 助手平台。这个平台要同时支持：

- 本地桌面能力
- 技能系统（skills）
- MCP 外部能力
- 后续可能加入的更多 agent 能力

整体设计遵循一条原则：

确定性执行与语义决策分层。

也就是：

- 路径访问、目录枚举、文件读取、权限边界等保持确定性
- 意图理解、能力选择、计划分解、摘要生成等允许使用 LLM

## 2. 分层架构

当前全局架构可以分为五层：

### 第一层：执行内核

- `graph`
- `runtime`

作用：

- 提供底层图执行、状态推进、路由和结构化 LLM 解析工具

### 第二层：基础能力层

- `tools`
- `memory`
- `policy`
- `observability`

作用：

- 工具注册与调用
- 会话/工作/索引记忆
- 权限与风险控制
- 执行过程观测

### 第三层：资源层

- `resources`

作用：

- 统一描述文件、目录、文档切片等资源对象
- 提供资源仓储与资源解析能力

### 第四层：应用层

- `applications`

作用：

- 将单域业务封装成 application
- 当前代表实现是 `desktop_content_application`

### 第五层：助手层

- `assistant`

作用：

- 维护会话
- 决定使用哪种 capability
- 统一接入 application、skill、MCP
- 形成单代理桌面 AI 助手的主循环

## 3. 关键对象关系

全局上最重要的对象关系如下：

### 资源与应用

- `ResourceRef`
- `ResourceRepository`
- `ResourceResolver`
- `ApplicationRunner`

应用层不直接操作裸路径，而是优先操作资源引用。

### 应用与助手

- `ApplicationSpec`
- `CapabilitySpec`
- `CapabilityRegistry`

应用层输出的是一个可注册 capability 的能力单元，而不是最终 assistant 自身。

### 助手与技能 / MCP

- `SkillRegistry`
- `MCPToolSpec`
- `MCPProvider`
- `MCPClientAdapter`
- `AgentLoop`

助手层统一对待：

- 本地 application capability
- skill capability
- MCP capability

## 4. capability 统一原则

系统当前明确采用 capability-first 的平台策略。

也就是任何可被代理选择和执行的东西，都应该尽量映射成 capability，包括：

- 本地桌面内容能力
- skills
- MCP tools
- 未来的 planner / report / search / automation 能力

采用这个策略的原因是：

1. selector 可以面对统一对象进行选择
2. policy 可以对 capability 做统一风控
3. observability 可以记录统一的选择历史
4. skills / MCP 不会成为框架中的旁路系统

## 5. LLM 在系统中的职责

当前系统中，LLM 主要承担以下职责：

- 结构化意图解析
- capability 选择
- stage parser 辅助解析
- 摘要与回复组织

LLM 当前不应承担：

- 文件系统真相判断
- 路径边界校验
- 本地读写执行
- 低层权限判定

这条边界是整个系统可控性的核心。

## 6. 中期演进方向

当前全局设计已经足够支撑下一阶段继续向真正的桌面 AI 助手演进。中期最值得推进的方向是：

1. planner / reviewer 的结构化 LLM 版本与更强循环策略
2. artifact / workspace / report 输出体系
3. 更真实的 MCP adapter 与 provider 生命周期管理
4. skill 的依赖建模与 planner 融合
5. capability selector 结合历史成功率、用户偏好与成本模型

目前第 2 章中助手层的这三项基础能力已经具备最小骨架：

- `plan -> act -> review -> continue/stop`
- `ApprovalManager / ApprovalRequest / ResumeToken`
- 扩展 capability metadata：`cost_hint`、`latency_hint`、`risk_class`、`dependency_readiness`、`output_type`

## 7. 非目标

当前设计仍然明确排除以下目标：

- 多代理协作
- 自动化 swarm
- 分布式调度
- GUI 桌面端完整产品壳
- 全自动高风险本地操作

这些能力不是不做，而是当前不应先做。
