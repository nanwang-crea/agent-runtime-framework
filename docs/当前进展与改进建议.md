# 当前进展与改进建议

## 1. 当前已经完成的内容

### 1.1 桌面内容 application 骨架

已经完成：

- `desktop_content_application`
- `run_stage_parser`
- `ResolverPipeline`
- `DesktopActionHandlerRegistry`

这意味着桌面内容能力已经不再是单文件硬编码流程，而是具备：

- 结构化阶段解析
- 模块化 resolver
- 模块化 action handler

### 1.2 资源、记忆、权限基础层

已经完成：

- `ResourceRef`
- `FileResource`
- `DirectoryResource`
- `DocumentChunkResource`
- `SessionMemory`
- `WorkingMemory`
- `IndexMemory`
- `SimpleDesktopPolicy`

这部分已经足够支撑一个受控的桌面内容助手。

### 1.3 单代理 assistant runtime

已经完成：

- `AssistantSession`
- `AssistantContext`
- `CapabilityRegistry`
- `AgentLoop`
- `SkillRegistry`
- `StaticMCPProvider`
- `MCPClientAdapter`

说明框架已经从“应用层骨架”进入“助手层骨架”阶段。

### 1.4 capability 语义增强

已经完成：

- `CapabilitySpec.description`
- `CapabilitySpec.safety_level`
- `CapabilitySpec.input_contract`
- `CapabilitySpec.cost_hint`
- `CapabilitySpec.latency_hint`
- `CapabilitySpec.risk_class`
- `CapabilitySpec.dependency_readiness`
- `CapabilitySpec.output_type`

并且这些元数据已经进入 capability selector prompt。

### 1.5 代理循环与审批恢复

已经完成：

- 最小 `plan -> act -> review -> continue/stop`
- `ExecutionPlan`
- `PlannedAction`
- `ApprovalRequest`
- `ApprovalManager`
- `ResumeToken`

说明 assistant runtime 已经不再只是单轮 dispatcher，而是具备了最小可恢复代理循环。

### 1.6 skills / MCP 第一阶段接入

已经完成：

- skill trigger phrases
- skill required capability hints
- skill planner hints
- discoverable MCP tool schema

说明 skills 和 MCP 已经不是单纯的占位插槽，而是开始进入统一 agent 框架。

### 1.7 可视化 demo 入口

已经完成：

- `DemoAssistantApp`
- 本地 HTTP demo server
- 内置 Web UI
- 会话历史与计划历史可视化

说明当前框架已经不只是库形态，开始具备可直接体验的最小演示入口。

### 1.8 模型接入层第一阶段

已经完成：

- `ModelRegistry`
- `ModelRouter`
- `ModelProfile`
- `AuthSession`
- `InMemoryCredentialStore`
- `OpenAICompatibleProvider`

并且 demo 已经支持：

- provider auth
- role 级模型选择
- conversation / selector / stage parser 使用模型路由
- 本地配置中心与默认路由持久化

### 1.9 Codex-style action runtime 第一阶段

已经完成：

- `agents/codex`
- `CodexTask`
- `CodexAction`
- `CodexActionResult`
- `VerificationResult`
- `CodexAgentLoop`
- 默认 codex tools
- 默认启发式 codex planner

并且已经支持：

- 通过 `call_tool` 读取工作区文件
- 通过 `call_tool` 列目录与总结文件
- 通过 `run_verification` 运行验证命令
- 通过 `apply_patch` 执行最小文本替换
- 通过 `move_path` / `delete_path` 执行移动与删除
- 通过 `create_path` / `edit_text` 执行创建与编辑
- artifact 持久化
- patch 类高风险动作的 approval / resume
- demo app 主链路已切到 `CodexAgentLoop`
- LLM-first next-action planner，规则 planner 退为 fallback
- demo / frontend shell 已经支持同会话切 agent / workspace 的 context 模型

### 1.10 Router / Evaluator 第二阶段

已经完成：

- `router` role
- `evaluator` role
- `LLM-first router + deterministic fallback`
- `LLM-first evaluator + deterministic fallback`
- evaluator 决策进入 `execution_trace`

说明当前 Codex runtime 已经不只是“选下一步动作”，而是开始具备：

- 普通对话与任务执行分流
- 任务结果是否足够完成目标的判断
- 不够时继续补工具调用或综合回答

## 2. 当前还没有完成的内容

### 2.1 artifact / workspace 输出体系

当前没有统一的产物管理，例如：

- 报告输出
- 计划输出
- 中间产物缓存
- 文件变更摘要

未来如果要做更完整的桌面 agent，这一层非常重要。

虽然 `CodexAgentLoop` 已经开始写入 artifact，但还没有形成完整的工作区级 artifact 体系，例如：

- 命令日志分类
- 文件 diff / patch artifact
- verification summary
- task completion report

### 2.2 真实 MCP 生命周期管理

当前 `MCPClientAdapter` 已经有了，但仍然是很薄的映射层，还没有：

- 会话级连接管理
- provider 健康状态
- tool 调用失败恢复
- schema 缓存

### 2.3 capability selector 仍偏轻量

虽然 selector 已经是 LLM-first，但还没有结合：

- 用户偏好
- 历史成功率
- capability 成本
- capability 风险等级
- capability 依赖状态

目前仍然属于“有语义，但还不够强”的阶段。

### 2.4 模型层仍然偏早期

当前虽然已经有独立模型层，但还没有真正的：

- 系统 keychain 级 credential store
- OAuth / browser login 流程
- provider 健康检查与配额状态
- 模型能力探测缓存
- 更丰富的多 provider 适配层

这意味着现在模型层已经脱离 demo 配置，但离产品级仍有明显距离。

## 3. 当前最应该继续改进的方向

### 优先级 1：把桌面能力继续下沉到 codex actions/tools

建议继续把当前 `desktop_content_application` 中的能力拆到 codex runtime，例如：

- create file / directory
- richer patch / diff apply
- rename / move directory
- delete directory with clearer guardrails
- multi-step workspace operations

原因：

只有这些能力真正落到 action/tool 层，Codex-style loop 才不会退化成“换了名字的 capability selector”。

当前 `desktop_content_application` 已经不再是 demo 产品入口，因此后续可以把它视为兼容层，按迁移节奏逐步收缩，而不是继续往里面加新能力。

当前代码也已经开始朝这个方向做最小迁移：

- `CapabilityRegistry` 继续保留，但开始支持 tool discovery entry
- tool 可以以 `tool:<name>` 形式进入能力目录，用于 selector / planner 的统一发现
- 这类条目默认不在 `AgentLoop` 中直接执行，而是标记为 `codex_only`
- `assistant` 可以通过桥接 capability 把任务执行委托给 `CodexAgentLoop`

这意味着 `assistant` 和 capability-center 仍然保留，但职责已从“未来唯一执行中心”转为“宿主层 + 发现层 + 兼容层”。

### 优先级 2：补真实 sandbox

建议下一步优先补：

- sandbox mode
- sandbox-aware command runner
- network policy
- sandbox denial 的结构化错误
- sandbox 状态进入 trace 和 UI

原因：

当前 router / planner / evaluator 的结构已经开始成熟，但执行隔离仍然不足。这会限制 Agent 的自治能力上限，也会让风险控制停留在“规则 + 审批”，还达不到主流 coding agent 的安全边界。

### 优先级 3：把 shell 从“选择器”推进到“profile workbench”

建议继续做：

- agent profile 级专属面板
- 同会话多 workspace 的更清晰上下文展示
- artifacts / approvals / run cards 按 agent profile 区分

原因：

现在前端已经具备 context 切换骨架，但还只是第一阶段 shell，不是完整的 agent workbench。

### 优先级 4：artifact 系统

建议补完整的工作区产物管理：

- 计划
- 报告
- 日志
- 摘要
- 工作区产物
- verification 证据

原因：

当前 codex runtime 已经开始生成 action 级 artifact，下一步应该把它提升到真正的 task/workspace 输出体系。

### 优先级 4：模型层产品化

建议继续补：

- 系统级 credential store
- OAuth / device code 登录
- provider health / quota / error state
- 更丰富的 provider 适配

原因：

模型层已经成型，但 auth 和 provider 生命周期还明显偏早期。

### 优先级 5：skills 与 planner 联动

建议后续让 skill 不只参与 fallback 触发，还参与：

- planner 候选生成
- capability ranking
- prompt assembly

### 优先级 6：真实 MCP 生命周期管理

建议继续补：

- 会话级连接管理
- provider 健康状态
- tool 调用失败恢复
- schema 缓存

## 4. 当前代码质量评价

优点：

- 架构层次已经逐渐清晰
- deterministic / LLM 职责边界合理
- 扩展点开始统一
- tests 覆盖持续跟上

问题：

- assistant 层目前还偏薄
- 双 runtime 并存期的概念成本开始上升
- 文档与代码虽然同步，但概念数量已经开始变多
- 一些 schema / metadata 仍然偏轻量
- Codex runtime 还没有形成真正的“生产级 agent loop”
- LLM 相关能力仍然缺少统一 provider/auth/router 基础设施

## 5. 结论

当前框架已经不是一个简单的桌面文件助手底座，而是一个正在成型的桌面 AI 助手平台骨架。

它现在最明显的优势是：

- 方向已经对了
- 扩展点已经开始统一
- skills / MCP / desktop capability 已经进入同一套编排结构

它现在最明显的短板是：

- 双层方向已经明确，但 codex runtime 还只是第一阶段
- 还没有形成统一 artifact / workspace 产物体系
- 模型层仍然偏早期
- planner 与 skills / MCP 仍然耦合较浅
- 会话级 MCP 生命周期治理仍偏薄

因此下一阶段最值得投入的，是把当前已具备循环和审批骨架的 assistant runtime，继续推进成“有产物、有模型基础设施、有更强规划能力”的代理系统。
