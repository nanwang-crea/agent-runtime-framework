# 当前进展与改进建议

## 1. 当前已经完成的内容

### 1.1 桌面内容 application 骨架

已经完成：

- `desktop_content_application`
- `run_stage_parser`
- `ResolverPipeline`
- `DesktopActionHandlerRegistry`

这意味着桌面内容能力已经不再是单文件硬编码流程，而是具备：

- 结构化阶段解析
- 模块化 resolver
- 模块化 action handler

### 1.2 资源、记忆、权限基础层

已经完成：

- `ResourceRef`
- `FileResource`
- `DirectoryResource`
- `DocumentChunkResource`
- `SessionMemory`
- `WorkingMemory`
- `IndexMemory`
- `SimpleDesktopPolicy`

这部分已经足够支撑一个受控的桌面内容助手。

### 1.3 单代理 assistant runtime

已经完成：

- `AssistantSession`
- `AssistantContext`
- `CapabilityRegistry`
- `AgentLoop`
- `SkillRegistry`
- `StaticMCPProvider`
- `MCPClientAdapter`

说明框架已经从“应用层骨架”进入“助手层骨架”阶段。

### 1.4 capability 语义增强

已经完成：

- `CapabilitySpec.description`
- `CapabilitySpec.safety_level`
- `CapabilitySpec.input_contract`

并且这些元数据已经进入 capability selector prompt。

### 1.5 skills / MCP 第一阶段接入

已经完成：

- skill trigger phrases
- skill required capability hints
- skill planner hints
- discoverable MCP tool schema

说明 skills 和 MCP 已经不是单纯的占位插槽，而是开始进入统一 agent 框架。

## 2. 当前还没有完成的内容

### 2.1 真正的 planner / reviewer / continue loop

当前虽然有 `AgentLoop`，但本质仍然是：

- 一次输入
- 一次 capability 选择
- 一次执行
- 一次输出

还没有形成真正的：

- plan
- act
- review
- continue

循环。

### 2.2 approval / resume 体系

当前只有底层 policy，没有真正的：

- 用户确认节点
- 中断恢复
- 需确认后继续执行

这会限制高风险桌面能力的扩展。

### 2.3 artifact / workspace 输出体系

当前没有统一的产物管理，例如：

- 报告输出
- 计划输出
- 中间产物缓存
- 文件变更摘要

未来如果要做更完整的桌面 agent，这一层非常重要。

### 2.4 真实 MCP 生命周期管理

当前 `MCPClientAdapter` 已经有了，但仍然是很薄的映射层，还没有：

- 会话级连接管理
- provider 健康状态
- tool 调用失败恢复
- schema 缓存

### 2.5 capability selector 仍偏轻量

虽然 selector 已经是 LLM-first，但还没有结合：

- 用户偏好
- 历史成功率
- capability 成本
- capability 风险等级
- capability 依赖状态

目前仍然属于“有语义，但还不够强”的阶段。

## 3. 当前最应该继续改进的方向

### 优先级 1：assistant loop 变成真正的 agent loop

建议下一步重点做：

- planner 节点
- review 节点
- continue / stop 判断

原因：

如果没有这层，当前框架更像“带 selector 的 dispatcher”，还不是一个真正强意义上的 agent。

### 优先级 2：approval / safety orchestration

建议增加：

- `ApprovalRequest`
- `ApprovalManager`
- `ResumeToken` 或等价恢复机制

原因：

桌面助手未来不可避免会碰到移动、重命名、整理等风险动作，这层必须尽快补上。

### 优先级 3：capability metadata 继续深化

建议后续补充：

- cost hint
- latency hint
- risk class
- dependency readiness
- output type

原因：

当前 selector 已经开始依赖 capability metadata，继续增强这层收益很高。

### 优先级 4：skills 与 planner 联动

建议后续让 skill 不只参与 fallback 触发，还参与：

- planner 候选生成
- capability ranking
- prompt assembly

### 优先级 5：artifact 系统

建议增加统一 artifact 层，管理：

- 计划
- 报告
- 日志
- 摘要
- 工作区产物

## 4. 当前代码质量评价

优点：

- 架构层次已经逐渐清晰
- deterministic / LLM 职责边界合理
- 扩展点开始统一
- tests 覆盖持续跟上

问题：

- assistant 层目前还偏薄
- 文档与代码虽然同步，但概念数量已经开始变多
- 一些 schema / metadata 仍然偏轻量
- 还没有形成真正的“生产级 agent loop”

## 5. 结论

当前框架已经不是一个简单的桌面文件助手底座，而是一个正在成型的桌面 AI 助手平台骨架。

它现在最明显的优势是：

- 方向已经对了
- 扩展点已经开始统一
- skills / MCP / desktop capability 已经进入同一套编排结构

它现在最明显的短板是：

- 还没有把单轮 dispatcher 进化成强 agent loop

因此下一阶段最值得投入的，是把 assistant runtime 从“可扩展调度器”继续推进成“真正的代理循环系统”。
