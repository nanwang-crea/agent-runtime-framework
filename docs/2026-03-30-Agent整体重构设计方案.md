## 目标

这份文档用于回答三个问题：

1. 当前 Agent 的端到端链路是否合理。
2. 参考更先进的工作型 Agent 结构，当前系统最需要重构的部分是什么。
3. 后续应该如何分阶段重构，而不是继续在局部补规则和补 prompt。

结论先行：

- 当前系统不是“方向错了”，而是“已经长到了必须做中层重构的阶段”。
- 当前骨架已经具备单代理工作流、工具执行、审批、恢复、记忆、run context 等基础能力。
- 真正影响“智能体感”的问题，不再是缺某个点状能力，而是任务语义、计划生成、证据判断、回答合成仍然分散在多个层级中重复判断。
- 后续最值得做的是从“多层各自猜一点”重构为“统一任务语义 + 状态驱动策略 + 统一证据管理 + 独立回答合成”。

- 已经从 capability-centric 转向 action/tool-centric。
- 已经有 run context、resource semantics、approval/resume、clarification、repair/recovery。
- planner 已支持 LLM-first，并增加了 MiniMax 非 JSON tool_call 容错与重试。

### 当前链路的核心问题

- route、profile、intent、plan 仍然是分层重复判断。

- 最终回答的 synthesizer 仍分散在 planner / evaluator / task_plans / direct respond 里。

### 2.5 shell 与 workspace tool 的边界

当前修正后的原则：

- `create_workspace_path / edit_workspace_text / apply_text_patch / move_workspace_path / delete_workspace_path` 仍然是优先策略。
- 但这不再意味着 shell 不能使用。
- `run_shell_command` 现在允许有限的 workspace 内文件命令，例如：
  - `touch`
  - `mkdir`
  - `cp`
  - `mv`
- 这些命令必须满足：
  - 路径在 workspace 内
  - 仍经过审批与 sandbox
- 因此系统不再把 Agent 写死成“创建文件绝不能用 touch”，而是改成：
  - 优先专用工具
  - shell 可用，但受 sandbox 和路径约束

## 3. 结构性问题清单

### 问题 1：缺少唯一任务语义入口

现状：

- `route`
- `profile`
- `intent`
- `task_plan`
- `evaluator`

都在做自己的部分判断。

影响：

- 同一句话在不同层可能被理解成不同任务。
- profile 选对了，不代表 plan 一定按同一语义生成。
- plan 做对了，不代表 evaluator 不会再按另一套逻辑补洞。

建议：

- 把 [semantics.py](/Users/munan/Documents/munan/my_project/ai/Agent_test/agent-runtime-framework/agent_runtime_framework/agents/codex/semantics.py) 升级成唯一上游 `TaskIntent` 入口。
- `profiles.py` 只消费 `TaskIntent.task_kind`，不再自己二次分类。
- `conversation.py` 的 route fallback 也只消费 `TaskIntent`。
- `task_plans.py` 和 `planner.py` 只使用 `TaskIntent`，不再额外做语义补推断。

### 问题 2：缺少 workspace-level task mode

现状：

- 已有 file / directory / resource semantics。
- 但“当前项目”“当前目录”“整个工作区”仍不是稳定的一等任务语义。

影响：

- `帮我列一下文件目录`
- `帮我生成一下该项目的摘要`
- `帮我列一下文件目录并帮我总结`

这类请求容易进入 clarification 或 resolve 误判。

建议：

- 在 `TaskIntent` 中新增：
  - `scope_kind = workspace_root | directory | file | module`
  - `default_scope = current_workspace`
  - `goal_mode = listing | overview | summary | read | explain | modify`

### 问题 3：`resolve_workspace_target` 被过度中心化

现状：

- 许多没有显式路径的请求，也先被丢给 `resolve_workspace_target`。

影响：

- resolve 失败时，整个任务很容易保守化成 clarification。
- 本该“默认当前项目”的任务，也被当成“缺少 target”。

建议：

- 把 `resolve_workspace_target` 降级成“显式 target / 模糊 target”场景的工具。
- 对于 `current workspace overview` 这种已知 scope，不再先 resolve。
- 让 planner 能直接对 workspace root 发起 `list / inspect`。

### 问题 4：`task_plans.py` 仍过于厚重

现状：

- 虽然已经接入 LLM-first plan builder，但 fallback 部分仍有较多 workflow if/else。

影响：

- plan 层仍然像半规则引擎。
- 工作流一多就会持续膨胀。

建议：

- `task_plans.py` 收缩为：
  - plan 数据结构
  - plan task status 推进
  - fallback 兜底策略
- 主工作流策略交给：
  - `TaskIntent`
  - `Plan Builder Prompt`
  - `Policy Validator`

### 问题 5：`evaluator.py` 还在做主流程补洞

现状：

- evaluator 不只是判断证据充分性，还在主动补：
  - inspect
  - rank
  - outline
  - synthesize

影响：

- planner 和 evaluator 的职责边界不清。
- 整体看起来像“计划没想完，靠 evaluator 临时补”。

建议：

- evaluator 收缩为：
  - `finish`
  - `continue`
  - `clarify`
  - `missing_evidence`
- 主流程里的标准动作链尽量前移到 planner / task_plan_builder。

### 问题 6：回答合成层没有独立出来

现状：

- synthesize 逻辑分散在：
  - planner 直接 respond
  - evaluator respond
  - task_plans synthesized_answer
  - direct output shortcut

影响：

- 回答风格不稳定。
- 同类任务的最终结构会因为中间路径不同而变化。

建议：

- 独立一个统一 `AnswerSynthesizer`
- 所有最终交付都从同一层生成
- planner / evaluator 只负责决定“是否该 synthesize”，不直接定义最终语言风格

## 4. 参考先进 Agent 后的目标结构

建议把当前架构重构成六层：

### 4.1 Intent Resolver

职责：

- 唯一任务语义入口
- 输出统一 `TaskIntent`

建议字段：

- `task_kind`
- `goal_mode`
- `scope_kind`
- `target_ref`
- `target_confidence`
- `needs_clarification`
- `expected_output`
- `allowed_strategy_family`

### 4.2 Task State

职责：

- 用统一状态表示任务进展

建议字段：

- `task_intent`
- `resolved_target`
- `resource_semantics`
- `evidence_items`
- `known_facts`
- `open_questions`
- `pending_actions`
- `plan_state`
- `confidence_state`

### 4.3 Policy / Plan Generator

职责：

- 基于 `TaskIntent + TaskState + ToolSet` 生成下一步 plan

原则：

- 模型优先生成 plan
- 本地只做安全校验、schema 校验、风险收敛

### 4.4 Action Executor

职责：

- 只执行，不猜语义

输出要求：

- `tool_output`
- `resource_semantics`
- `evidence_kind`
- `changed_paths`
- `verification_result`

### 4.5 Evidence Manager

职责：

- 判断当前证据是否足够支撑目标
- 决定还缺哪类证据

不做：

- 不承担主流程编排
- 不直接定义最终回答文案

### 4.6 Answer Synthesizer

职责：

- 统一最终回答
- 统一回答结构和风格

建议输出模式：

- 目录列举
- 目录概览
- 项目摘要
- 文件讲解
- 修改总结
- 验证报告

## 5. 推荐的重构路线

### 第一阶段：统一任务语义

目标：

- 所有入口只生成一次 `TaskIntent`

需要做的内容：

- 重构 `semantics.py`
- `profiles.py` 改为只消费 `TaskIntent`
- `conversation.py` route fallback 改为只消费 `TaskIntent`
- 新增 `workspace_overview / workspace_listing / project_summary` 三类 workspace-level 语义

验收：

- `帮我列一下文件目录`
- `帮我列一下文件目录并帮我总结`
- `帮我生成一下该项目的摘要`

以上请求都默认指向当前 workspace root

### 第二阶段：瘦身 task_plans

目标：

- `task_plans.py` 不再写主工作流

需要做的内容：

- 保留 plan data structure
- 保留 action/plan status 推进
- 删除大部分主 workflow if/else
- 用 plan builder prompt + policy validator 替代

验收：

- repository overview / file reader / change and verify 的主链路能由模型 plan 稳定给出

### 第三阶段：把 evaluator 收缩成证据判断器

目标：

- evaluator 只负责判断证据，而不是补主流程

需要做的内容：

- 输出统一 evidence sufficiency decision
- 将标准动作链前移到 planner / plan builder
- synthesize 从 evaluator 中拆出去

验收：

- 同类任务的动作链更稳定
- 回答风格更统一

### 第四阶段：独立 Answer Synthesizer

目标：

- 所有最终回答都走统一出口

需要做的内容：

- 提供 `answer_mode`
- 为不同任务类型定义稳定输出模版
- 将当前散落在 planner/evaluator/task_plans 的 respond 逻辑收敛

验收：

- 目录类回答、文件类回答、修改类回答的风格稳定

## 8. 最终判断

如果一句话概括当前状态：

- 当前系统已经是“具备正确骨架的 Agent runtime 原型”
- 但还不是“经过统一中层语义和状态机打磨的工作型 Agent”

所以后续最重要的不是继续补规则，而是做一次围绕以下三点的整体重构：

- 统一任务语义
- 统一状态驱动决策
- 统一证据与回答合成

只要这三层真正统一，当前框架的“智能感”会比继续补局部规则提升明显得多。
