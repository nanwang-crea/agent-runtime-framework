# Agent Runtime Framework 分层设计草案

## 1. 背景

当前系统已经具备以下核心能力：

- 基于 `WorkflowNode` / `GraphExecutionRuntime` 的原子执行能力
- 基于 `AgentGraphState` 的多轮规划、判断、恢复与持久化
- 基于 `CapabilitySpec` / `CapabilityMacro` 的初步能力抽象
- 基于 `judge`、`subgraph_planner`、`capability_diagnosis` 的动态工作流生成能力

随着系统复杂度增加，当前实现中逐渐出现以下问题：

- 部分高层任务语义直接落在原子节点或 planner 中，职责边界不够清晰
- planner 仍较多直接面向 executor/node 级别规划，而不是先选择能力与成熟配方
- “读文件前先定位目标”“修改前先收集证据”“修改后再验证”这类高频套路尚未被系统化沉淀为统一的可复用结构
- 原子工具、能力语义、工作流编排三层之间仍存在耦合，后续扩展容易继续把复杂度堆到 planner 或节点实现中

因此，需要一套明确的分层设计，将“怎么执行”“解决什么问题”“通常怎么组合”“何时选择哪条链路”拆开。

---

## 2. 设计目标

本方案的目标是：

1. 将最小执行单元与任务语义解耦
2. 将高频任务套路沉淀为可复用配方，而不是每轮从零规划
3. 让 planner 优先面向 capability / recipe，而不是直接拼 node 序列
4. 让 capability diagnosis 能输出“缺哪种能力”，而不是只停留在单个 executor 失败
5. 保留原子执行层的细粒度与权限边界，不把底层做粗做死
6. 为后续 capability extension、治理、自动恢复提供稳定的分层接口

---

## 3. 非目标

本方案不打算：

- 替换现有 `GraphExecutionRuntime` / `AgentGraphRuntime`
- 废除原子 `WorkflowNode` / executor 体系
- 把所有行为封装成少量“大工具类”
- 一次性重写 planner、judge、capability diagnosis 全链路
- 改变现有外部 API 契约

---

## 4. 设计原则

### 4.1 原子层只做执行，不做策略判断
原子 executor 的职责应限制为：

- 是否可执行
- 执行结果是什么
- 执行失败的技术原因是什么

不承担以下职责：

- 这种任务通常应该先做什么
- 当前是否该走这条策略
- 是否应该切换到其它任务链路

### 4.2 能力层表达“任务能力”，而不是“工具名”
能力的命名与语义应面向用户目标与任务产出，而不是底层 executor 名称。

正确方向：

- `locate_workspace_target`
- `read_target_content`
- `collect_workspace_evidence`
- `edit_target_file`

不推荐方向：

- `search_then_read`
- `call_content_search_and_chunked_file_read`
- `apply_patch_workflow`

### 4.3 配方层表达“成熟套路”，不是强制流程
高频任务应该被沉淀为默认 recipe / macro，但 recipe 是：

- 默认路径
- 可裁剪
- 可跳步
- 可在失败后切换

不是固定死顺序。

### 4.4 planner 优先选择语义单元，再展开为执行图
规划过程应从：

- 直接输出 node 序列

演进为：

- 先选 capability / recipe
- 再展开为 node / edge 子图
- 最后由 runtime 执行

### 4.5 运行层负责执行与恢复，不决定任务语义
`AgentGraphRuntime` / `GraphExecutionRuntime` / `AgentGraphState` 应主要负责：

- 状态管理
- 调度执行
- 恢复与校验
- 持久化与观测

而不是承担“高层任务建模”。

---

## 5. 建议分层

## 5.1 原子执行层

### 定义
系统中最小的可执行单元，对应当前 `WorkflowNode` / executor / tool 调用。

### 典型组件
- `interpret_target`
- `target_resolution`
- `workspace_discovery`
- `content_search`
- `chunked_file_read`
- `tool_call`
- `create_path`
- `write_file`
- `append_text`
- `apply_patch`
- `move_path`
- `delete_path`
- `verification`
- `verification_step`
- `clarification`

### 职责
- 执行单步动作
- 产生结构化结果
- 记录失败信息与恢复线索
- 不对多步链路做封装判断

### 设计约束
- 输入输出必须明确
- 错误信息可被上层诊断消费
- 不依赖 recipe 语义
- 尽量保持可组合、可测试、可替换

---

## 5.2 能力层

### 定义
能力层表达系统“能解决哪类问题”，是对一组原子节点能力的语义抽象。

### 推荐能力类型
- `locate_workspace_target`
- `collect_workspace_evidence`
- `read_target_content`
- `edit_target_file`
- `apply_structural_change`
- `verify_workspace_change`
- `diagnose_verification_failure`

### 能力对象建议字段
在现有 `CapabilitySpec` 基础上，建议逐步增强：

- `capability_id`
- `description`
- `intents`
- `preconditions`
- `produces`
- `toolchains`
- `failure_signatures`
- `verification_recipe`
- `extension_policy`

### 含义说明

#### `preconditions`
表示启用该能力前应满足的条件，例如：

- 已确认目标
- 已具备候选路径
- 已有足够证据
- 已存在修改后的文件

#### `produces`
表示该能力成功后可为后续提供的结构化状态，例如：

- `resolved_target`
- `interpreted_target`
- `evidence_items`
- `chunks`
- `modified_paths`
- `verification_result`

### 能力层作用
- 作为 planner 的中间语义层
- 作为 capability diagnosis 的推荐对象
- 作为 recipe 的构建模块
- 作为扩展治理的基本单位

---

## 5.3 配方层

### 定义
配方层表示“某类任务通常如何由多个能力组合完成”。

配方不是原子执行单元，也不是底层工具链；它是系统沉淀下来的“成熟工作套路”。

### 当前适合沉淀的配方
- `resolve_then_read_target`
- `search_then_read_evidence`
- `locate_inspect_edit_verify`
- `inspect_patch_verify_python`
- `resolve_then_move_or_rename`
- `resolve_then_delete_path`

### 配方对象建议字段
建议将当前 `CapabilityMacro` 演进为更强的 recipe 结构：

- `recipe_id`
- `description`
- `intent_scope`
- `required_capabilities`
- `optional_capabilities`
- `entry_conditions`
- `exit_conditions`
- `fallback_recipes`
- `verification_strategy`

### 示例

#### 读类配方
`resolve_then_read_target`

- entry:
  - 用户目标可能歧义
- required capabilities:
  - `locate_workspace_target`
  - `read_target_content`
- exit:
  - 已拿到与目标相关的 `chunks` / `facts`

#### 改类配方
`locate_inspect_edit_verify`

- entry:
  - 用户要求修改工作区内容
- required capabilities:
  - `locate_workspace_target`
  - `collect_workspace_evidence`
  - `edit_target_file`
  - `verify_workspace_change`
- exit:
  - 已完成修改并拿到验证结果

### 配方层作用
- 提供默认成熟路径
- 提升 planner 稳定性
- 降低每轮自由规划成本
- 提高 capability diagnosis 的解释性

---

## 5.4 规划层

### 定义
规划层根据用户目标、当前状态、历史失败信息、能力视图与治理约束，决定下一步使用哪个 capability / recipe，并展开为子图。

### 建议职责拆分

#### A. 任务意图归类
输入：

- 用户目标
- `GoalEnvelope`
- 当前 `AgentGraphState`

输出：

- 任务形态
- 是否需要 target resolution
- 是否需要 read / edit / verify

#### B. 前置条件分析
根据当前状态判断是否缺：

- 明确目标
- 证据
- 修改上下文
- 验证方案

#### C. recipe / capability 选择
优先级建议：

1. 命中成熟 recipe
2. 选择 capability chain
3. 最后才回退自由 node 规划

#### D. 子图展开
将已选 capability / recipe 转成 `PlannedNode` + `WorkflowEdge`

### 推荐新增模块
- `workflow/planning/capability_selection.py`
- `workflow/planning/recipe_expansion.py`

### 与现有模块关系
- `judge`：继续负责判断是否接受、是否重规划、允许哪些方向
- `capability_diagnosis`：负责识别缺口与推荐能力
- `subgraph_planner`：逐步从“直接生成 node 图”演进为“基于 recipe / capability 展开图”

---

## 5.5 运行层

### 定义
运行层是系统的执行与状态基础设施，不承担高层任务语义。

### 关键对象
- `WorkflowRuntimeContext`
- `AgentGraphState`
- `WorkflowRun`
- `GraphExecutionRuntime`
- `AgentGraphRuntime`

### 职责
- 执行 workflow graph
- 管理多轮迭代状态
- 记录过程事件
- 处理审批、澄清、恢复、持久化
- 不负责决定任务能力与组合套路

### 当前建议
继续保留：

- 原子 graph 调度能力
- `AgentGraphState` 作为图级状态本体
- `WorkflowRuntimeContext` 作为执行期依赖与只读快照容器

---

## 6. 关键对象关系

### 6.1 从用户目标到执行的推荐路径

1. 用户输入形成 `GoalEnvelope`
2. 规划层分析 intent 与当前 state
3. capability / recipe selector 选择一条语义链路
4. recipe expander 展开成 `PlannedSubgraph`
5. `GraphExecutionRuntime` 执行原子节点
6. `AgentGraphState` 累积证据、失败、恢复与记忆
7. `judge` 决定是否结束或继续下一轮

### 6.2 状态关系

- `WorkflowRuntimeContext`
  - 执行期上下文
  - application/workspace/policy/process/agent_graph_state
- `AgentGraphState`
  - 图级长期状态
  - iteration / judge_history / failure_history / memory_state
- `WorkflowRun`
  - 一次具体图执行的运行载体
  - 当前子图中间结果与 node 状态

---

## 7. 推荐命名规范

### 原子层
动词 + 技术动作

- `content_search`
- `chunked_file_read`
- `apply_patch`

### 能力层
动词 + 任务语义对象

- `locate_workspace_target`
- `read_target_content`
- `edit_target_file`

### 配方层
动词链路 + 任务目标

- `resolve_then_read_target`
- `locate_inspect_edit_verify`
- `inspect_patch_verify_python`

### 避免
- 在能力层直接写 executor 名称
- 在 recipe 层直接表达底层工具细节
- 在 planner 中直接硬编码固定 executor 顺序作为策略

---

## 8. 推荐迁移路线

## 8.1 第一阶段：语义补全
在 `CapabilitySpec` 中增加：

- `preconditions`
- `produces`

保持现有 `toolchains` 不动。

## 8.2 第二阶段：宏升级为 recipe
把 `CapabilityMacro` 从简单 `capability_chain` 升级为：

- `entry_conditions`
- `required_capabilities`
- `optional_capabilities`
- `exit_conditions`

## 8.3 第三阶段：规划先选 recipe
将 `subgraph_planner` 中的部分策略逻辑拆分为：

- capability selection
- recipe expansion

## 8.4 第四阶段：恢复/诊断对齐到 capability 层
让恢复策略和 capability diagnosis 输出更稳定的“能力缺口”，减少直接依赖原子节点名。

---

## 9. 与当前代码的对应关系

### 当前已经具备的雏形
- `CapabilitySpec`
- `CapabilityMacro`
- `capability_diagnosis`
- `subgraph_planner`
- `judge`
- `AgentGraphState`

### 当前主要缺口
- capability 缺乏明确前置条件与产出契约
- macro 仍偏简单链表
- planner 仍偏 node-first，而不是 recipe-first
- 高层语义与原子节点存在一定混写

---

## 10. 设计决策总结

本方案采用如下架构决策：

1. 保留原子 executor，不将其粗暴合并为少量大工具
2. 将 capability 作为任务语义层
3. 将 recipe 作为成熟套路沉淀层
4. 将 planner 从 node-first 演进为 capability/recipe-first
5. 将 runtime 保持为执行与恢复基础设施
6. 通过 capability diagnosis 将失败信息提升到能力语义层

---

## 11. 后续实施建议

优先级建议如下：

1. 增强 `CapabilitySpec` 字段
2. 演进 `CapabilityMacro` 为 recipe 结构
3. 增加 capability selection / recipe expansion 模块
4. 逐步将 `subgraph_planner` 改为先选 recipe 再展开 node
5. 调整 diagnosis / recovery 的输出语义，向 capability 层对齐

---

## 12. 开放问题

以下问题可在实施前进一步确认：

1. capability 是否需要显式声明 `consumes` / `produces`
2. recipe 是否允许条件分支与 fallback
3. planner 选 recipe 时，是否需要显式优先级与评分模型
4. capability extension 是扩展能力定义，还是扩展 recipe 模板
5. diagnosis 输出是否直接面向 capability id，还是同时保留 node 级别线索
