# 通用 Agent 改造清单

## 目标

把当前 Agent 从“关键词命中 + 特例工作流”改造成“基于任务语义、资源语义和状态驱动决策”的通用 Agent。

当前的重点不是继续为单个问题补 prompt 或补规则，而是补齐一层稳定的中间语义，使目录理解、文件阅读、代码库讲解、写操作和错误恢复都能走统一机制。

## 当前核心问题

1. 任务理解过度依赖脆弱关键词，表达一变化就掉到错误 profile。
2. 资源解析只能定位路径，不能把“目录 / 文件 / 工作区”提升成后续规划约束。
3. planner 还是正则触发器，不是基于任务状态和资源状态决策。
4. evaluator 只是在补特例，缺少统一的证据充分性判断。
5. 工具层错误会直接暴露给用户，没有进入可恢复的内部控制流。
6. 测试集偏 benchmark 句式，缺少真实自然表达覆盖。

## 总体改造方向

### 1. 统一任务语义层

目标：
- 统一表达用户到底是在聊天、讲解代码库、查看目录、读文件，还是修改工作区。

需要落地的能力：
- 把 route / profile / planner 中分散的关键词判断收敛成统一任务语义。
- 让语义层输出稳定字段，例如：
  - `task_profile`
  - `goal_mode`
  - `target_hint`
  - `target_kind`
  - `needs_clarification`

验收标准：
- “这个目录下面都是在讲什么”
- “帮我过一下这个 package”
- “这里主要做什么”
- “看看这个模块都有哪些文件”

以上表达都能进入一致的目录 / 代码库理解链路。

### 2. 统一资源语义层

目标：
- 资源解析不再只返回路径，而要返回资源类型及后续允许动作。

需要落地的能力：
- 让 resolver 输出目录 / 文件 / 工作区等语义类型。
- 把“当前目录”“下面”“这里”“刚才那个文件”等引用统一建模。
- 为后续 planner 提供 `allowed_next_actions` 一类的约束信息。

验收标准：
- 目录不会再被当作文件摘要目标直接执行。
- follow-up 引用不再依赖零散字符串判断。

### 3. 重写 planner 为状态驱动决策

目标：
- planner 不再主要依赖正则，而是基于任务语义、资源语义和已收集证据决定下一步。

需要落地的能力：
- 目录理解任务固定走：
  - locate
  - list
  - inspect
  - read entrypoint if needed
  - synthesize
- 文件阅读任务固定走：
  - locate
  - read / summarize
  - synthesize
- 写操作任务固定走：
  - locate
  - preview
  - approval
  - apply
  - verify
  - synthesize

验收标准：
- planner 能对同一意图的不同说法给出稳定动作序列。

### 4. 建立统一错误恢复层

目标：
- 常见错误不再直接终止任务，而是变成 planner 可恢复的控制流。

需要落地的能力：
- `RESOURCE_IS_DIRECTORY` 自动转入目录 inspect / list。
- 目标不存在时优先尝试候选匹配或澄清。
- 目标歧义时优先给候选列表，而不是误命中。
- shell / tool 失败进入结构化恢复分支，而不是只吐异常。

验收标准：
- 用户看到的是“我改用目录讲解方式继续处理”，不是直接 error card。

### 5. 建立统一证据充分性判断

目标：
- evaluator 不再只补单个特例，而是判断当前证据是否足够回答目标。

需要落地的能力：
- 判断是否只拿到了目录列表，还没拿到结构说明。
- 判断是否需要补读 README、`__init__.py`、入口文件或关键模块。
- 判断是否已经可以给出总结，还是应该继续探索。

验收标准：
- 最终回答不再只是工具原文直出，而是基于证据生成稳定总结。

### 6. 扩充测试体系

目标：
- 测试不再只覆盖 benchmark 句式，而要覆盖真实用户表达和恢复路径。

需要补的测试：
- 自然语言目录讲解请求
- 模糊引用和 follow-up
- 目录误走文件工具后的自动恢复
- 多候选目标澄清
- evaluator 的继续探索判断

验收标准：
- 真实表达改写后仍然通过，而不是只对固定文案通过。

## 分阶段实施

### 第一阶段：目录理解链路语义化

目标：
- 先修掉最明显的“不智能”表现，让目录 / 代码库理解任务更稳定。

实施内容：
- 扩大 `repository_explainer` 的自然语言识别覆盖。
- 提升目标路径抽取能力，支持中英混杂表达。
- 在 `read/summarize` 打到目录时自动恢复到目录 inspect。

当前状态：
- 已开始实现。

### 第二阶段：资源语义外显化

目标：
- 让 resolver 输出不仅有路径，还有资源类型和允许动作。

实施内容：
- 为 resolver 增加资源语义结构。
- 在 plan metadata 中显式传播 target kind。
- 减少工具层内部的重复判断。

### 第三阶段：planner 状态机化

目标：
- 让 planner 真正基于状态，而不是基于字符串规则。

实施内容：
- 统一目录理解、文件阅读、写操作三条标准策略。
- 把 evaluator 的补洞逻辑逐步回收到 planner / plan expander。

### 第四阶段：通用恢复和澄清

目标：
- 让常见失败进入恢复或澄清，而不是报错结束。

实施内容：
- 目标不存在、歧义、越界、工具失败、验证失败等都统一纳入恢复策略。

## 本次已启动的改造

1. 扩大了目录 / 代码库理解任务的语义识别范围。
2. 提升了混合自然语言中的目标目录抽取能力。
3. 为目录误走 `read_workspace_text` / `summarize_workspace_text` 增加了自动恢复。
4. 为 resolver 增加了显式资源语义，能够区分 `file` / `directory` 及其允许动作。
5. 将资源语义传播到了 Codex plan state，使 `gather_context` 可以按目标类型选择 `list` 或 `read`。
6. 将目录类继续探索判断从工具名特例，收敛为基于资源语义和证据充分性的 evaluator 逻辑。
7. 在 `CodexAgentLoop` 中加入了统一的低风险重试机制：`retriable` 的低风险动作会自动重试，避免临时失败直接终止任务。
8. 将 `retriable` 的 `AppError` 收敛为结构化失败结果，并接入 plan 流转，而不是直接把异常抛到用户面前。
9. 为 plan 增加了通用 `recover_failed_action` 扩展点，允许在动作失败后由模型插入一个恢复动作，再继续后续 synthesize。
10. 修正了恢复任务插入时的依赖传播，避免恢复任务错误地继续依赖已失败节点。
11. 将 `index_memory` 从简单 KV 缓存扩展为可检索的结构化记忆条目，并支持相关性搜索。
12. 把工作区焦点摘要沉淀为 `workspace_focus` 记忆，并接入 `resolve_workspace_target` 的解析链路，使历史记忆开始真正影响当前目标定位。
13. 在任务完成后，把 `final_output`、`known_facts`、`typed_claims` 沉淀成 `task_conclusion` / `workspace_fact` 记忆，使任务级知识开始跨轮可检索。
14. 在目标解析时引入 session-aware follow-up 处理，像“继续说刚才那个模块”这类表达会优先命中当前会话焦点，而不是回退到工作区根目录。
15. 为资源解析增加了显式 `resolved / ambiguous / unresolved` 状态，并把 session 焦点、index 记忆、资源语义合并进同一条解析链路。
16. 当 locate 阶段遇到多候选目标时，Agent 不再误命中，而是转成 `clarify_target` 响应，明确要求用户指定候选。
17. 为 change/create 任务保留了“目标尚不存在也可继续”的豁免，避免新建文件场景被误判成需要澄清。
18. 为长期记忆增加了 OpenClaw 风格的 Markdown 记录化落盘实现，`remember/search` 在不引入 embedding 的前提下具备可审阅、可重载的持久化基础。
19. 将 Markdown 记录化记忆从 demo 初始化推广到了更通用的 `ApplicationContext` 默认入口，只要存在工作区根目录就会默认落到 `.arf/memory.md`。
20. 将 `clarify_target` 从一次性文本回复升级为可恢复的 clarification loop；当任务进入目标澄清状态时，下一轮输入会优先续跑原任务，而不是新起一个无上下文的任务。
21. 为 clarification loop 增加了跨重启恢复能力：待澄清任务状态会持久化到工作区侧的 runtime state 中，应用重启后仍能继续原任务，而不是退化成普通聊天。

下一步优先做：
1. 把可检索记忆继续扩展到用户偏好、任务约束和稳定规则，而不只存工作区知识。
2. 把 follow-up 引用继续从“刚才那个/这个模块”扩展到更复杂的跨轮指代和多实体引用。
3. 让 clarification loop 支持取消、改问新任务和多轮候选收缩，而不只覆盖单次目标补充。
4. 继续压缩 evaluator 中剩余的工具名特例，收敛到统一状态机。
5. 把“重试同一动作”“插入恢复动作”“目标澄清”继续合并成更完整的失败处理状态机，而不只覆盖当前的单步恢复插入。

## 对标主流 Agent 的下一步清单

### P0：直接影响“智力感”的系统层

1. 可检索的长期记忆层
   目标：让 Agent 不只记住当前 turn，而是能把历史事实、路径、用户偏好、任务结论沉淀成可查询记忆。
2. codebase indexing 与语义检索
   目标：让 Agent 能跨目录、跨文件、跨历史上下文地找相关代码，而不只靠路径和关键词。
3. follow-up / 指代消解 / 歧义澄清
   目标：稳定处理“这个目录”“下面那个”“刚才那个文件”“继续说这个模块”。
4. 统一 planner / evaluator / retry / recover 状态机
   目标：让下一步决策、证据充分性、失败恢复都在同一状态模型里完成。
5. 长任务 context compaction
   目标：长链路任务中能阶段性压缩上下文，保留关键事实，避免越做越笨。

### P1：显著提升任务完成率的执行层

1. 后台异步任务执行
2. 多 agent / 并行探索 / 隔离工作区
3. 环境快照与依赖准备
4. 验证-修复-复验闭环
5. 面向仓库历史、PR、测试结果的证据检索

### P2：提升体验和可运营性的产品层

1. 用户可审阅的 memory / rules 演化机制
2. 更强的 trace 可解释性
3. 更自然的中间反馈与任务阶段展示
4. 自动化技能装配与任务模板沉淀

## 当前执行优先级

当前先做 `P0-1：可检索的长期记忆层` 的最小闭环：
1. 先把 `index_memory` 从简单 KV 缓存扩展为“结构化记忆条目 + 查询接口”。
2. 再把工作区焦点、路径摘要、任务结论沉淀为可检索记忆，并逐步转成可审阅的 Markdown 记录。
3. 然后让 `resolve_workspace_target` 和后续 planner 能消费这些记忆，开始真正影响任务行为。
