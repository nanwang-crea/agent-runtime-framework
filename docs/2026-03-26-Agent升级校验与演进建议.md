# Agent Runtime Framework 升级校验与演进建议

## 目标

这份文档先做两件事：

1. 校验 [Agent优先级改造清单](./Agent优先级改造清单.md) 与当前代码是否一致。
2. 结合 `OpenCode` 与 `OpenClaw` 的公开能力，重新梳理下一阶段更值得投入的 Agent 升级方向。

---

## 一句话结论

原清单的总体方向是对的，但它已经部分过时。

当前代码并不是“还没有运行时骨架”，而是已经具备了：

- 基础版统一 system prompt 入口
- task profile 分类
- task-level state machine 骨架
- resource semantics 传播
- evidence insufficiency / recovery / clarification loop

真正还薄弱的地方，已经不再是“有没有这些概念”，而是：

- run context 还不够厚，缺 instruction inheritance 和 loaded-instructions tracking
- agent profile 还不是权限驱动的稳定画像
- tool runtime 还没有统一校验、repair、标准化错误出口
- history / status / long-session runtime 还不够产品化
- subagent 仍未 session 化

所以，下一阶段不建议再把重点放在“planner 继续加规则”，而应该把系统升级成更像一个 `runtime OS for agent` 的结构。

## 对原优先级的修正

原文最需要修正的，不是方向，而是“紧迫度判断”。

### 已经不应再排成最高优先级的项

- “把高频任务状态机化”
  - 不是没做，而是已经做了第一版。
  - 现在更该做的是把更多任务接入同一状态机模型，而不是继续讨论要不要状态机化。
- “让 resource semantics 成为 planner 一等输入”
  - 这件事已经进入主链路。
  - 下一步应该是让它从 `Codex 专项能力` 升级为 `assistant/runtime 通用能力`。

### 应该上移的项

- instruction inheritance
- run context builder
- permission-driven agent profiles
- history/status packet
- tool runtime repair

原因很简单：

- 现在系统最影响体感的问题，已经不是“完全不会规划”，而是“每轮掌握的信息还不够完整、角色边界还不够稳、失败兜底还不够成熟”。

## 从 OpenCode / OpenClaw 借鉴什么

## OpenCode 最值得借鉴的点

基于 2026-03-26 可公开访问文档，`OpenCode` 的强项主要不在“某个 planner 更聪明”，而在运行时工程。

### 1. Agent 是稳定画像，不只是任务标签

`OpenCode` 文档明确有 primary agents 和 subagents，并内置 `Build / Plan / general / explore / compaction / summary` 等角色，同时 agent 权限可以独立覆盖。

这意味着：

- agent 本身就带行为边界
- prompt、permission、mode 是一起生效的
- “探索 / 规划 / 修改 / 压缩上下文”被拆成不同工作角色

这比当前系统的 `task_profile` 更厚一层。

### 2. instruction 继承是主能力，不是补丁

`OpenCode` 官方规则文档明确支持：

- 项目根 `AGENTS.md`
- 全局 `~/.config/opencode/AGENTS.md`
- `CLAUDE.md` fallback
- `opencode.json` 里的额外 instruction 文件和远程 URL

这一层直接决定了模型开局是否“知道仓库怎么协作”。

### 3. 权限模型直接参与 Agent 建模

`OpenCode` 权限文档明确支持：

- 全局 permission
- per-agent permission override
- 对 `task`、`bash`、`edit` 等动作做 allow / ask / deny

这意味着它不是只有一个 policy，在执行时才临时拦；而是 agent 出场时就已经被画像化。

### 4. subagent 是 child session，不是普通内部调用

`OpenCode` agent 文档明确把 subagent 当成独立 child session，并支持在 parent / child session 间导航。

这对复杂任务很关键，因为：

- 子任务上下文隔离
- 父任务不会被子任务污染
- 子任务可追踪、可恢复、可回看

### 5. 长会话靠系统 agent 维护

`OpenCode` 官方 agent 文档明确有隐藏的 `compaction`、`summary`、`title` agent。

这说明它把长对话管理当成运行时能力，而不是临时压缩字符串。

## OpenClaw 最值得借鉴的点

基于 2026-03-26 可公开访问文档，`OpenClaw` 的强项更偏“厚工作区上下文”和“Markdown 化长期记忆”。

### 1. workspace bootstrap files 直接进首轮上下文

`OpenClaw` 文档明确会在新 session 首轮注入：

- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `USER.md`
- `IDENTITY.md`
- `BOOTSTRAP.md`

这会让 agent 一上来就处于一个“有人设、有规则、有工具说明、有用户画像”的环境里。

### 2. Memory 是可审阅的 Markdown，而不是黑盒 embedding

`OpenClaw` 官方 memory 文档明确把记忆放在：

- `memory/YYYY-MM-DD.md`
- `MEMORY.md`

并提供 `memory_search`、`memory_get` 工具。

这和当前仓库 `MEMORY.md` 里的偏好方向是一致的，也比直接上 embedding 更适合当前阶段。

### 3. 工具说明与 workspace 说明解耦

`OpenClaw` 把 `TOOLS.md` 当成“如何使用工具的用户约定”，而不是工具注册表本身。

这很值得借鉴，因为它意味着：

- 工具实际能力由 runtime 决定
- 工具使用风格可以由项目或用户自己定制
- prompt-facing guidance 可以脱离代码发布节奏更新

## 深一层判断：我们的 Agent 应该怎么升级

如果综合看当前代码现状、`OpenCode` 和 `OpenClaw` 的做法，我建议把下一阶段升级目标定义成：

### 从“有 planner 的单 Agent”升级成“有运行时操作系统的工作型 Agent”

这句话的含义是：

- 不再把“变聪明”主要寄托在 planner prompt 上
- 而是让模型每一轮都拿到更厚、更稳定、更结构化的工作上下文
- 再用清晰角色、工具 runtime、session runtime 去放大模型能力

这条线会比继续堆 planner 规则更稳，也更接近成熟产品。

---

## 建议的升级主线

## 主线一：先补 Run Context Builder

这是最优先的主线。

建议新增统一的 `run context builder`，由它负责为 planner、evaluator、conversation、classifier 生成同构上下文包，至少包括：

- identity / active agent
- workspace / cwd / sandbox / writable roots
- permission snapshot
- loaded instructions
- focused resources
- recent completed actions
- current user message
- pending clarification / pending verification
- current plan state
- memory snapshot
- available tools summary

这一步完成后，当前很多“模型没看全”的问题会立刻下降。

## 主线二：把 profile 从 task profile 升级成 runtime persona

建议从当前的 `task_profile` 继续升级出真正的 agent persona：

- `build`
- `plan`
- `explore`
- `general`
- `summary`
- `compaction`

每个 persona 同时控制：

- prompt 模板
- tool allowlist / ask / deny
- 是否允许写
- 默认 step budget
- 是否允许 spawn subagent
- 默认的 evidence threshold

`task_profile` 仍然有价值，但它应该退到 persona 内部的任务模式，而不是承担全部角色边界。

## 主线三：把 Tool Runtime 做成统一稳定层

建议把现有 `execute_tool_call()` 升级成真正的 runtime wrapper，统一处理：

- input schema validation
- 参数别名修复
- 工具名大小写修复
- invalid-tool fallback
- 长输出截断 metadata
- 标准错误结构
- tool retry / backoff
- tool call trace id

这里是最容易用工程方法换体感的地方。

## 主线四：把 History/Status 做成显式数据结构

现在的 follow-up context 还偏轻。

建议新增统一的 `session status packet`，明确区分：

- current message
- relevant history summary
- recent actions
- focused resources
- pending questions
- pending approvals
- pending clarifications
- plan progress
- session summary

这样“继续上一步”“刚刚那个文件”“顺便看看那里”这类任务会明显稳很多。

## 主线五：把 subagent 做成 child session

这一步不要太早做，但一定要做对。

建议直接按 child session 设计，而不是普通函数式 subtask：

- 有父子关系
- 有独立 memory / plan / artifacts
- 有 task_id / session_id
- 可恢复
- 可汇总回父任务

这会是后面处理复杂编码任务、批量改造、并行验证的关键能力。

## 主线六：把 long-session 维护独立成系统能力

建议在 subagent 之前就先设计接口，但在 subagent 之后再落成熟实现。

建议目标：

- `summary agent`
- `compaction agent`
- `session status agent`

不要一开始就做复杂自动压缩策略，先把“何时触发、输出什么结构、如何回填主 session”定义清楚。

## 调整后的优先级建议

### P0

- 统一 run context builder
- instruction 自动发现、继承、loaded-instructions tracking
- tool runtime wrapper 2.0
- permission-driven agent persona

### P1

- history/status packet
- 扩展 task plan state machine 到更多任务类型
- 把 resource semantics 推到 assistant/runtime 通用层
- structured output final path

### P2

- child-session subagent runtime
- summary / compaction / status system agents
- 更完整的 long-memory workflow

## 一个更务实的实施顺序

### 第 1 轮：先补“模型每轮看到什么”

- run context builder
- instruction inheritance
- loaded instruction tracking
- history/status packet

预期收益：

- 模型不再只站在局部 prompt 上思考
- 代码仓任务的“懂上下文”体感会立即提升

### 第 2 轮：再补“模型怎么被约束成不同工作角色”

- agent persona
- permission snapshot
- tool runtime wrapper 2.0
- structured output final path

预期收益：

- 行为边界更稳
- 工具调用更少翻车
- 总结与结构化输出更一致

### 第 3 轮：最后补“复杂任务和长会话”

- child-session subagent
- summary / compaction / status
- memory workflow 升级

预期收益：

- 复杂任务拆分明显更强
- 长 session 不容易越聊越乱
- 更接近成熟 coding agent 产品体验

---

## 最终建议

如果只允许我给一个核心建议，那就是：

### 下一阶段不要再以 planner 为中心设计 Agent，而要以 runtime 为中心设计 Agent。

当前框架已经证明：

- 你有不错的 task-plan 和 resource-semantics 基础
- 你不是没有 Agent 骨架

接下来真正能把体感拉开层级的，是这四件事：

1. 更厚的 run context
2. 更稳定的 agent persona
3. 更成熟的 tool runtime
4. 更产品化的 session/subagent 机制

只要这四件事做对，当前 Agent 会从“有时挺聪明的执行框架”升级成“持续稳定、像在真正工作”的 Agent。
