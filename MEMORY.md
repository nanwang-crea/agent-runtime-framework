## 用户偏好

- [2026-03-18] 当用户描述“希望 AI 理解是否需要列出某个文件夹下的内容” -> 优先落实为基于语义/LLM 的意图判断与资源定位，不要把需求降级成关键字硬匹配 -> 验收标准：默认方案先检查并启用 LLM-first 路由，只有在模型不可用时才保留规则兜底。
- [2026-03-25] 当扩展 agent 的计划、动态插 task 或能力路由时 -> 默认采用模型优先的决策与扩展方式，不要先上规则模板；规则只能作为模型不可用时的兜底 -> 验收标准：新增能力优先接入 LLM 决策点，代码中能明确看出 model-first 路径。
- [2026-03-25] 当扩展 agent 的长期记忆层时 -> 优先采用 OpenClaw 风格的 Markdown / 记录化记忆，不要优先引入 embedding 依赖 -> 验收标准：记忆能力的下一步演进先走结构化记录、可审阅和规则化检索路径。
- [2026-03-25] 当用户明确要求“不要最小改动，要做完善修改”且能力涉及跨 turn / 跨重启状态时 -> 默认实现为持久化、可恢复的设计，不接受只存在于进程内的临时状态方案 -> 验收标准：关键状态在应用重启后仍可恢复，相关测试覆盖 restart 场景。

## 协作规则

- [2026-03-18] 当用户指出“你理解错了需求” -> 先复述纠错点，再直接按新理解重做，不延续旧实现方向 -> 验收标准：本轮改动里能看到需求纠偏对应的代码或验证。
- [2026-03-25] 当 agent 输出结构化总结或任务总结时 -> 需要附带引用来源，不只给结论 -> 验收标准：最终回答包含“引用”区段，并列出实际使用的路径或命令来源。
- [2026-03-25] 当已经有明确设计清单和实施路径时 -> 直接连续执行后续批次，不要每完成一小层就停下来向用户要确认；只有遇到真实阻塞、破坏性分歧或高风险选择时才中断 -> 验收标准：后续回合默认继续实现、验证和汇报，而不是把正常推进变成确认点。

## 易错点与修正

- [2026-03-18] 处理桌面内容请求时，把“列某个目录/子目录内容”的需求误收敛成关键字匹配 -> 需要同时检查 capability 选择、intent 解析和资源定位是否仍被规则短路 -> 验收标准：至少验证 capability selector 与 desktop intent parser 的执行顺序符合 LLM-first 设计。
